这个文件记录我在 Phase 1 的数据理解阶段发现的数据问题和注意点。

1. 哪些地方以后可能会算错；
2. 为什么会算错；
3. 我在 SQL 和 Python 分析时准备怎么避免这些错误。

## 3. 数据质量问题、我的理解与后续处理方式

|    编号 | 发现的问题                      | 真实检查结果                                                                                     | 解释                                                     | 初步处理结果                                                                                | 是否影响核心分析 |
| ----: | -------------------------- | ------------------------------------------------------------------------------------------ | --------------------------------------------------------------- | -------------------------------------------------------------------------------------- | -------- |
| DQ-01 | 一笔订单可能有多个商品。               | `order_items.order_id` 有 13,984 条重复记录；但 `order_id + order_item_id` 组合没有重复。                 | 这不是错误数据，而是一笔订单里可以买多个商品。所以同一个订单号会在商品明细表里出现好几次。                   | 以后算“订单数量”时，不能直接数商品明细表有多少行。我会保证同一个 `order_id` 只算一次。只有分析商品金额、商品品类时，才保留每一件商品的明细。          | 是        |
| DQ-02 | 一笔订单可能有多条支付记录。             | `order_payments.order_id` 有 4,446 条重复记录；但 `order_id + payment_sequential` 组合没有重复。          | 有些订单可能用了多次付款、分期付款，或有不止一条支付记录。                                   | 核心分析先不直接使用支付表。如果后面要分析支付方式，我会先把同一订单的支付记录整理成一条，再和订单表连接。                                  | 是        |
| DQ-03 | 评价表中，同一个评价 ID 或订单 ID 可能重复。 | `review_id` 有 814 条重复；`order_id` 有 551 条重复。                                                | 我不能直接假设“一笔订单只有一条评价”。如果直接连接，某些订单可能会被重复计算。                        | 现在先不删除评价记录。以后做评分分析前，我会先检查重复评价具体是什么情况，再决定同一订单应该保留哪条评价或怎么汇总。                             | 是        |
| DQ-04 | 有些订单没有实际送达时间。              | `order_delivered_customer_date` 缺失 2,965 条，占订单表 2.98%。                                     | 没有实际送达时间，我就不知道它到底花了几天送到，也无法判断它有没有晚到。                            | 以后算配送天数和延迟率时，只使用已经送达、而且实际送达时间和预计送达时间都不为空的订单。                                           | 是        |
| DQ-05 | 有些订单缺少审批时间或交给承运商的时间。       | `order_approved_at` 缺失 160 条；`order_delivered_carrier_date` 缺失 1,783 条。                    | 这些字段和订单审批、卖家发货流程有关，但不是我核心版最主要的配送指标。                             | 核心版暂时不使用这两个字段计算主要指标，所以不需要删除订单。以后如果分析“审批慢不慢”或“发货慢不慢”，再只看这些时间完整的订单。                      | 否        |
| DQ-06 | 有些商品没有品类名称。                | `products.product_category_name` 缺失 610 条，占商品表 1.85%。                                      | 这些商品真实存在，只是原始数据没有告诉我它属于哪个品类。不能因为没有品类就把订单删掉。                     | 做品类分析时，我会把这些商品暂时标记为 `Unknown` 或 `Missing category`。图表里会保留这个类别，并说明它代表原始数据缺失。            | 是        |
| DQ-07 | 少量商品品类没有英文翻译。              | 32,341 条非空品类中，有 13 条不能匹配英文翻译表；匹配率为 99.96%。                                                 | 大部分葡萄牙语品类都有英文翻译，只有极少数没有。                                        | 做英文图表时，能翻译的就显示英文；不能翻译的品类保留原来的葡萄牙语名称，或标记为 `Untranslated`。不因为没有翻译就删除它们。                  | 是        |
| DQ-08 | 很多评价没有标题或正文。               | `review_comment_title` 缺失 87,656 条，占 88.34%；`review_comment_message` 缺失 58,247 条，占 58.70%。 | 很多客户只打了分，没有写文字。这不代表他们满意，也不代表他们不满意。                              | 核心版不做评论文本分析，只使用 `review_score` 做平均评分和低评分率分析。                                           | 否        |
| DQ-09 | 地理位置表中的邮编前缀不唯一。            | `geolocation_zip_code_prefix` 有 981,148 条重复记录。                                             | 同一个邮编前缀在地理位置表里会出现很多次，所以它不能直接当成“一对一”的主键使用。                       | 核心版的州级分析直接使用客户表里的 `customer_state`，暂时不依赖地理位置表。如果以后做地图，再先想清楚同一个邮编前缀的多条记录要怎么合并。          | 否        |
| DQ-10 | 少量客户和卖家的邮编无法在地理位置表中找到。     | 客户邮编有 278 条未匹配，匹配率 99.72%；卖家邮编有 7 条未匹配，匹配率 99.77%。                                         | 大部分邮编都能找到位置，但有少量找不到。                                            | 核心版不靠经纬度计算关键指标，所以暂时不受影响。如果以后做地图，找不到位置的记录不放进地图，并在说明中写清楚地图不是 100% 覆盖。                    | 否        |

## 6. 后续分析中遵守的规则

1. 我不会修改或删除原始 CSV 文件。
2. 我不会因为看到缺失值就直接把整行数据删掉。
3. 算订单数量时，要记住：**一笔订单只能算一次。**
4. 商品金额和品类分析可以保留商品明细，因为一笔订单里本来就可能有多个商品。
5. 客户复购必须使用 `customer_unique_id`，不能使用 `customer_id`。
6. 配送分析只使用已经送达、并且送达时间完整的订单。
7. 没有评论正文或没有评价，不代表客户满意。
8. 核心版州级分析优先使用 `customer_state`，不急着做复杂地图。
9. 所有结果都来自 2016—2018 年的公开历史数据，不代表 Olist 当前情况。
10. 后面每做一个处理决定，都要记录原因，不能为了让图表更好看就随便删数据。
