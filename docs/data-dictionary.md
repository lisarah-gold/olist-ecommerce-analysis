# 数据字典（Data Dictionary）

## 1. 文档目的

本文件用于说明 Olist 巴西电商公开数据集中各张核心表的：

* 一行数据代表什么；
* 关键字段是什么；
* 表与表之间如何连接；
* 字段的原始读取类型、缺失情况和唯一性检查结果；
* 后续项目中这些字段会被用来做什么。

本文件的检查结果来自：

```text
notebooks/01_data_understanding.ipynb
```

数据质量问题、后续如何避免算错，以及具体处理规则，记录在：

```text
docs/data-quality-log.md
```

---

## 2. 数据集范围

* 数据来源：Brazilian E-Commerce Public Dataset by Olist
* 原始数据位置：`data/raw/`
* 原始 CSV 数量：9 张
* 订单下单时间范围：2016-09-04 21:15:19 至 2018-10-17 17:30:18
* 实际送达时间最晚为：2018-10-17 13:22:46
* 预计送达时间最晚为：2018-11-12 00:00:00

* 项目分析主题：

  * 月度经营趋势
  * 商品品类表现
  * 客户重复购买
  * 区域差异
  * 配送表现
  * 配送与客户评分的关系

> 注意：本项目使用的是历史公开数据。后续所有结论只说明这段历史样本中的现象，不代表 Olist 当前的真实经营情况。

## 4. 表关系与分析粒度说明

### 4.1 核心表关系

```text
customers.customer_id
        ↓
orders.customer_id

orders.order_id
        ↓
order_items.order_id
        ↓
products.product_id

order_items.seller_id
        ↓
sellers.seller_id

orders.order_id
        ↓
order_payments.order_id

orders.order_id
        ↓
order_reviews.order_id

products.product_category_name
        ↓
product_category_translation.product_category_name
```

### 4.2 已验证的关联覆盖情况

下面的检查表示：子表中的非空连接字段，能否在对应父表中找到匹配值。

这说明这些表可以用于后续分析中的连接；但这不代表数据库里已经创建了正式的外键约束，也不代表所有连接都是一对一关系。


| 连接关系                                                                                  |    匹配率 | 当前结论                              |
| ------------------------------------------------------------------------------------- | -----: | --------------------------------- |
| `orders.customer_id → customers.customer_id`                                          |   100% | 每一笔订单都能找到对应的客户记录。                 |
| `order_items.order_id → orders.order_id`                                              |   100% | 每一件商品明细都能找到对应订单。                  |
| `order_payments.order_id → orders.order_id`                                           |   100% | 每一条支付记录都能找到对应订单。                  |
| `order_reviews.order_id → orders.order_id`                                            |   100% | 每一条评价记录都能找到对应订单。                  |
| `order_items.product_id → products.product_id`                                        |   100% | 每一件订单商品都能找到对应商品。                  |
| `order_items.seller_id → sellers.seller_id`                                           |   100% | 每一件订单商品都能找到对应卖家。                  |
| `products.product_category_name → product_category_translation.product_category_name` | 99.96% | 绝大多数非空品类都能找到英文翻译；有 13 条非空品类记录未匹配。 |
| `customers.customer_zip_code_prefix → geolocation.geolocation_zip_code_prefix`        | 99.72% | 大部分客户邮编可找到地理位置，但地理位置表不是一对一关系。     |
| `sellers.seller_zip_code_prefix → geolocation.geolocation_zip_code_prefix`            | 99.77% | 大部分卖家邮编可找到地理位置，但地理位置表不是一对一关系。     |

## 5. 重要分析粒度规则

| 表                              | 数据粒度      | 我需要记住什么                                                  |
| ------------------------------ | --------- | -------------------------------------------------------- |
| `orders`                       | 订单粒度      | 一行就是一笔订单，订单数量通常从这里计算。                                    |
| `order_items`                  | 订单商品明细粒度  | 一笔订单可以买多个商品，所以同一 `order_id` 会出现多次。                       |
| `order_payments`               | 支付记录粒度    | 一笔订单可能有多次付款或多条支付记录。                                      |
| `order_reviews`                | 评价记录粒度    | 有些订单可能对应多条评价，不能默认一笔订单只有一个评分。                             |
| `customers`                    | 客户与订单关联粒度 | `customer_id` 用来连接订单；分析真实客户是否复购时使用 `customer_unique_id`。 |
| `products`                     | 商品粒度      | 一行代表一个商品。                                                |
| `sellers`                      | 卖家粒度      | 一行代表一个卖家。                                                |
| `geolocation`                  | 地理位置记录粒度  | 一个邮编前缀会对应多条位置记录，不能直接当成主键。                                |
| `product_category_translation` | 品类翻译粒度    | 一行代表一个葡萄牙语品类名对应的英文名称。                                    |

# 6. 各表字段说明

## 6.1 customers：客户与订单关联表

### 表说明

* 原始文件：`olist_customers_dataset.csv`
* 行数与列数：99,441 行 × 5 列
* 一行代表什么：一条客户与订单关联记录。
* 主要作用：把订单连接到客户所在城市、州和邮编前缀；使用 `customer_unique_id` 识别同一个真实客户是否多次购买。
* 关键提醒：`customer_id` 和 `customer_unique_id` 不一样。

| 字段                         | 字段含义        | 原始读取类型  | 缺失情况 | 是否用于连接 | 本项目用途         | 已验证结果与备注                          |
| -------------------------- | ----------- | ------- | ---- | ------ | ------------- | --------------------------------- |
| `customer_id`              | 客户记录 ID     | `str`   | 无缺失  | 是      | 连接 `orders` 表 | 无重复，可作为本表的唯一标识；订单表连接匹配率为 100%。    |
| `customer_unique_id`       | 同一真实客户的唯一标识 | `str`   | 无缺失  | 否      | 客户重复购买分析      | 用于把同一个真实客户的多笔订单归到一起；不是订单表的直接连接字段。 |
| `customer_zip_code_prefix` | 客户邮编前缀      | `int64` | 无缺失  | 可选     | 区域补充、可选地图分析   | 与地理位置表匹配率为 99.72%；地理位置表中该字段不唯一。   |
| `customer_city`            | 客户所在城市      | `str`   | 无缺失  | 否      | 城市补充分析        | 核心版不以城市作为主要统计维度。                  |
| `customer_state`           | 客户所在州       | `str`   | 无缺失  | 否      | 州级订单、配送和评分分析  | Q4 区域差异分析的核心字段。                   |

# 7. Phase 1 已验证的重要结论

完成 `01_data_understanding.ipynb` 后，已经确认：

1. 原始数据中预期的 9 张 CSV 文件都存在，没有缺失文件或额外 CSV 文件。
2. `orders.order_id`、`customers.customer_id`、`products.product_id` 和 `sellers.seller_id` 都无缺失、无重复。
3. `order_items` 是商品明细表，一笔订单可能有多件商品。
4. `order_payments` 是支付记录表，一笔订单可能有多条支付记录。
5. `order_reviews` 中不能直接假设一笔订单只对应一条评价。
6. `products.product_category_name` 有 610 条缺失，需要在品类分析中单独标记。
7. 所有订单、商品明细、支付记录、评价记录和卖家记录都能找到对应的核心表记录。
8. `geolocation_zip_code_prefix` 不唯一，因此地理位置表不能直接作为严格的一对一连接表。
9. 订单日期字段可以成功转换为日期时间格式，没有非空值转换失败。
10. 订单数据的核心分析范围将主要使用 `order_status = 'delivered'` 的已完成订单。
