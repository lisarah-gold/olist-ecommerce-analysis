## 1. 这份文件的作用

这份 `project-charter.md` 是本项目的总规则和阶段路线图。它的作用不是把项目变成一份很难的商业咨询报告，而是帮助项目始终围绕一个现实目标推进：

> **完成一个真实、完整、可解释、可放进 GitHub 和简历的数据分析作品集，证明项目作者会使用 SQL、Python、数据可视化和基础业务分析方法。**

它会影响项目的总体方向，但不会把项目“锁死”。项目推进过程中可以根据真实数据调整图表、字段、分析顺序和可选内容；不过，数据来源、项目真实性、核心技能目标和核心交付物应保持稳定。

### 1.2 本项目最重要的判断标准

本项目不追求“术语很多”或“模型很复杂”。每完成一个部分，都优先判断：

1. 这个分析问题是否真的能用现有数据回答？
2. 计算过程是否正确、可以复查？
3. 我是否能用自己的话讲清楚它？
4. 它是否能展示 SQL、Python 或可视化能力？
5. 它是否值得放进最终作品集？

只要这五点成立，项目就是有效的。复杂不等于优秀；**真实、清晰、可解释**才更适合数据分析实习作品集。

## 2. 项目概述

本项目使用 **Olist Brazilian E-Commerce Public Dataset**，完成一次从公开原始数据理解、MySQL 数据导入与查询、SQL 多表分析、Python 数据清洗与可视化，到 Tableau Dashboard、项目报告和 GitHub 展示的端到端数据分析实践。

项目将遵循以下工作流：

```text
业务问题
→ 数据表理解
→ 数据质量检查
→ MySQL 导入与 SQL 验证
→ Python 清洗、分析与可视化
→ Dashboard 与简要报告
→ GitHub 整理与面试表达准备
```

本项目不要求像资深电商分析师一样提出复杂经营战略。它要求的是：

- 看懂多张真实业务数据表；
- 正确连接数据并避免重复计算；
- 用 SQL 算出基础指标；
- 用 Python 清洗、统计和画图；
- 用简单但专业的语言解释结果；
- 把过程整理成他人能看懂的项目作品。

## 3. 项目定位与边界

### 3.1 本项目是什么

这是一个基于公开历史数据完成的独立数据分析作品集项目。它重点展示：

- 数据理解能力；
- MySQL 与 SQL 查询能力；
- Python/pandas 数据清洗与分析能力；
- 可视化和 Dashboard 表达能力；
- 基础业务问题拆解能力；
- GitHub 项目整理与复现能力。

### 3.2 本项目不是什么

本项目不应被描述为：

- Olist 官方委托项目；
- 真实企业实习项目；
- Olist 当前经营情况分析；
- 中国电商市场研究；
- 营销增长、广告投放或用户获客分析；
- 销售预测、利润预测或机器学习项目；
- 能证明“平台一定应该这样决策”的咨询报告。

### 3.3 为什么不强调“用户增长”

Olist 数据包含订单、客户、商品、卖家、支付、评价和配送等信息，但没有完整的流量、广告投放、获客渠道、营销活动、成本、利润和库存字段。

因此，本项目将重点放在数据能够可靠支持的内容：

> **经营趋势、商品品类、客户重复购买、区域差异、配送表现和客户评分。**

这比“用户增长”更准确，也更容易在面试中解释，不会因为概念过大而被追问到数据无法证明的内容。

- 本项目使用的指标口径。

### 4.3 必须写在 README、报告和 Dashboard 中的边界说明

### 4.4 结论表达规则

| 可以这样说 | 不应这样说 |

| “在本数据样本中，延迟订单的低评分比例更高。” | “延迟配送一定导致低评分。” |
| “该品类在已完成订单中贡献较高的商品金额代理指标。” | “该品类为平台创造了最高利润。” |

## 6. 核心数据表与必须理解的关系

| 数据表主题 | 典型内容 | 本项目主要用途 |
|---|---|---|
| 客户表 | 客户 ID、唯一客户 ID、州、城市、邮编前缀 | 识别客户、重复购买和区域信息 |
| 订单表 | 订单状态、下单时间、发货时间、实际/预计送达时间 | 经营趋势、配送时间、订单状态 |
| 订单商品表 | 商品、卖家、商品价格、运费 | 商品金额、品类、卖家和订单明细 |
| 支付表 | 支付方式、支付金额、分期次数 | 支付结构和金额核验 |
| 评价表 | 评分、评论创建时间、回复时间 | 客户体验和低评分分析 |
| 商品表 | 商品类别、重量、尺寸等 | 商品品类及基础商品特征 |
| 卖家表 | 卖家 ID、州、城市、邮编前缀 | 可选的卖家和区域分析 |
| 地理位置表 | 邮编前缀、城市、州、经纬度 | 区域补充和可选地图 |
| 类别翻译表 | 商品分类中英文映射 | 提高图表和报告可读性 |

### 6.1 必须真正理解的三件事

1. **`customer_id` 与 `customer_unique_id` 不一样。**  
   `customer_id` 用于连接订单；`customer_unique_id` 才能识别同一个真实客户是否多次购买。

2. **一个订单可能有多件商品。**  
   订单表是一笔订单一行；订单商品表可能是一笔订单多行。做 `JOIN` 前要先确定你是按订单算，还是按商品明细算。

3. **一个订单可能有多条支付记录。**  
   如果订单表直接连接支付表，订单数可能被重复。计算订单数量时必须使用 `COUNT(DISTINCT order_id)` 或先汇总支付表。

### 6.2 本项目的分析粒度规则

| 问题类型 | 优先粒度 | 例子 |
|---|---|---|
| 订单趋势、配送、评分 | 订单粒度 | 一个订单只算一次 |
| 商品金额、品类、卖家 | 订单商品明细粒度 | 一个订单内的每个商品都要保留 |
| 重复购买 | 唯一客户粒度 | 使用 `customer_unique_id` |
| 支付方式 | 支付记录或订单汇总粒度 | 先说明多条支付记录如何处理 |

## 7. 指标口径初版：只需要理解，不需要死背术语

项目开始时先使用以下定义。真正开始计算后，需要在 `docs/metric-definitions.md` 中补充对应表、字段、SQL 和筛选条件。

| 普通说法 | 项目中使用的指标名 | 初版定义 | 注意事项 |
|---|---|---|---|
| 完成了多少订单 | 已完成订单数 | `order_status = 'delivered'` 的去重订单数 | 一个订单只能算一次 |
| 商品卖出了多少钱 | 商品金额代理指标 | 已完成订单的 `order_items.price` 总和 | 不等同于真实营收或利润 |
| 每笔订单平均买了多少钱商品 | 平均订单商品金额 | 商品金额代理指标 ÷ 已完成订单数 | 需先按订单正确汇总 |
| 有多少人再次购买 | 复购客户数 | 至少有 2 笔已完成订单的 `customer_unique_id` 数 | 不能直接用 `customer_id` |
| 再次购买的人占多少 | 复购率 | 复购客户数 ÷ 至少 1 笔已完成订单的客户数 | 必须写明时间范围 |
| 发货到送达用了多久 | 配送时长 | 实际送达时间 − 下单时间 | 仅对已完成且时间完整订单计算 |
| 有没有晚到 | 延迟订单 | 实际送达时间晚于预计送达时间 | 不是“发货慢”的同义词 |
| 晚到订单占多少 | 延迟率 | 延迟订单数 ÷ 可判断是否延迟的订单数 | 分母不包含时间缺失订单 |
| 差评占多少 | 低评分率 | 评分 ≤ 2 的评价订单 ÷ 有效评价订单 | 没评价不等于好评 |

### 7.1 “代理指标”是什么意思

“代理指标”不是故弄玄虚。它只是提醒我们：数据里有 `price`，但没有企业的完整收入、退款、成本和利润信息，所以不能把 `price` 总和直接叫做“公司真实营收”。

在报告中可用的简单表达是：

> 本项目使用已完成订单中商品价格之和作为商品金额代理指标，用于比较不同月份、品类和区域的相对表现；该指标不代表企业真实财务收入或利润。

## Phase 0：项目初始化与规则建立

### 阶段目标

建立一个干净、可运行、不会丢文件的项目环境。这个阶段不下载数据、不做分析、不写业务结论。

### 为什么先做这一步

初学项目最常见的问题不是不会分析，而是文件混乱、环境混乱、代码找不到、数据误上传、做完不知道如何展示。Phase 0 是为了把这些麻烦提前解决。

### 具体任务

#### 0.1 创建项目根目录

1. 在本地创建项目文件夹，例如：`olist-ecommerce-analysis`；
2. 用 VS Code 打开整个文件夹，不要只打开某一个文件；
3. 在根目录放入本文件：`project-charter.md`。

#### 0.2 创建 GitHub 仓库

1. 在 GitHub 创建同名仓库；
2. 建议仓库设置为 Public，便于未来作为作品集展示；
3. 先不要上传数据文件；
4. 将本地仓库与 GitHub 仓库连接。

#### 0.3 创建基础文件

本阶段只需要创建：

```text
README.md
.gitignore
requirements.txt
AGENTS.md
docs/phase-checklist.md
docs/learning-log.md
```

#### 0.4 配置 Python 环境

1. 在项目根目录创建 `.venv`；
2. 激活虚拟环境；
3. 安装最基础依赖，例如 `pandas`、`numpy`、`matplotlib`、`seaborn`、`jupyter`、`sqlalchemy`、`pymysql`；
4. 把已安装依赖写入 `requirements.txt`；

## Phase 1：下载数据、理解表结构与建立数据字典

### 阶段目标

真正看懂数据由哪些表组成、每张表记录什么、表与表如何连接。此阶段仍然以“理解”为主，不急于得出业务结论。

### 为什么这一步重要

数据分析项目中，很多错误来自“以为自己知道表是什么意思”。尤其 Olist 是多表数据：订单、订单商品、客户和支付记录的行数都不一样。没有理解数据表，后面的 SQL 很容易重复计算。

### 具体任务

#### 1.1 下载和保存原始数据

1. 手动下载 Olist 原始 CSV；
2. 解压到本地 `data/raw/`；
3. 检查文件数量、文件名和大小是否合理；
4. 不把原始 CSV 提交到 GitHub。

#### 1.2 列出核心表

先把每张 CSV 的文件名记录到 `docs/data-dictionary.md`。不需要一开始把所有字段背下来。

建议按下列顺序理解：

1. 订单表；
2. 客户表；
3. 订单商品表；
4. 评价表；
5. 支付表；
6. 商品表；
7. 商品分类翻译表；
8. 卖家表；
9. 地理位置表。

#### 1.3 为每张表回答四个问题

对每张核心表，用自己的话回答：

1. 一行代表什么？
2. 这张表的主键或唯一标识是什么？
3. 它能和哪张表连接？靠什么字段连接？
4. 它未来能回答什么问题？

示例：

| 表 | 一行代表什么 | 关键字段 | 可以回答什么 |
|---|---|---|---|
| 订单表 | 一笔订单 | `order_id` | 订单状态、下单时间、送达时间 |
| 订单商品表 | 订单中的一个商品明细 | `order_id`、`product_id` | 商品金额、品类、卖家 |
| 客户表 | 一个订单对应的客户记录 | `customer_id`、`customer_unique_id` | 客户地区、重复购买 |

#### 1.4 建立数据字典

在 `docs/data-dictionary.md` 中，每张表至少记录：

| 字段 | 字段含义 | 数据类型 | 是否可能缺失 | 是否用于连接 | 本项目用途 |
|---|---|---|---|---:|---|

不需要解释所有边缘字段。优先写本项目会使用的字段。

#### 1.5 做第一轮数据检查

可以用 Python Notebook 或先用 CSV 预览完成以下检查：

- 每张表有多少行、多少列；
- 每个字段的数据类型看起来是否合理；
- 主键是否重复；
- 哪些关键字段为空；
- 日期字段的最早和最晚时间；
- 订单状态有哪些取值；
- 商品类别是否有缺失；
- 评价是否所有订单都有。

#### 1.6 绘制表关系说明

不要求画复杂 ER 图。先用一个简单关系图或 Markdown 文本说明即可：

```text
customers -- customer_id --> orders -- order_id --> order_items
                                  ├--> order_payments
                                  └--> order_reviews
order_items -- product_id --> products
order_items -- seller_id --> sellers
```

#### 1.7 记录数据质量问题

在 `docs/data-quality-log.md` 中按下表记录：

| 发现日期 | 问题 | 涉及表/字段 | 初步处理方式 | 是否影响核心指标 |
|---|---|---|---|---:|

示例问题：商品类别缺失、评价缺失、送达时间缺失、一个订单多条支付记录、城市名写法不一致等。

### 本阶段需要理解的内容

完成 Phase 1 后，你应能不用看答案解释：

- `customer_id` 和 `customer_unique_id` 为什么不同；
- 一个订单和一个订单商品明细为什么不是一回事；
- 为什么一个订单连接订单商品表后可能变成多行；
- 为什么评价缺失不能直接当作“客户满意”；
- 为什么数据检查要在分析之前完成。

## Phase 2：MySQL 建库、CSV 导入与数据验证

### 阶段目标

### 为什么这一步重要

Python 可以直接读 CSV，但 SQL 是数据分析实习常见能力。通过 MySQL 导入和查询，你会练习真实工作中常见的流程：看表、关联、检查数据、写指标 SQL、导出结果。

## Phase 3：SQL 核心分析

### 阶段目标

使用 SQL 回答核心版六个问题，并让每一个结果都有清晰的计算逻辑、过滤条件和解释。

### 为什么这一步重要

这是项目最能证明 SQL 能力的阶段。面试官不只想看你会写 `SELECT`，还会关注：你是否理解多表连接、是否能避免重复计算、是否能把业务问题拆成可计算指标。

### 工作方式

所有 SQL 可以先写在一个文件中：

```text
sql/05_core_business_analysis.sql
```

每个问题使用固定格式：

```sql
/*
问题：
分析粒度：
统计范围：
核心口径：
预期输出：
*/

-- SQL 查询
```

### 具体任务

#### 3.1 Q1：月度经营趋势

**目的：** 看不同月份的订单数量和商品金额是否有变化。

**需要计算：**

- 月份；
- 已完成订单数；
- 商品金额代理指标；
- 平均订单商品金额。

**建议方法：**

1. 从订单表筛选已完成订单；
2. 用订单商品表汇总每笔订单商品价格；
3. 再按下单月份汇总；
4. 确保订单数使用去重订单数。

**需要输出：** 月度汇总表。

**你需要能解释：**

- 为什么只分析已完成订单；
- 为什么商品金额从订单商品表计算；
- 为什么平均订单商品金额不能直接用商品行的平均价格代替。

#### 3.2 Q2：商品品类表现

**目的：** 看哪些类别更常被购买、商品金额较高，以及是否有评分偏低的类别。

**需要计算：**

- 品类名称；
- 已完成订单数；
- 商品金额代理指标；
- 平均每笔订单商品金额；
- 平均评分或低评分率；
- 评价覆盖订单数。

**建议做法：**

- 先做订单商品与商品类别的连接；
- 再与订单表筛选已完成订单；
- 与评价表连接前先确认评价粒度为订单；
- 对评价覆盖太少的品类，在报告中保持谨慎。

**需要输出：** Top 10 品类表，以及“订单量高但评分较低”的候选品类表。

**你需要能解释：**

- 订单量与商品金额为什么是两种不同表现；
- 为什么评分结果需要说明评价覆盖情况；
- 为什么“评分低”不能直接说明产品质量差。

#### 3.3 Q3：客户重复购买

**目的：** 展示客户层面的基础分析能力，但不把它包装成复杂“增长模型”。

**需要计算：**

- 至少有一笔已完成订单的唯一客户数；
- 购买 1 次的客户数；
- 购买 2 次及以上的客户数；
- 重复购买客户占比；
- 重复购买客户贡献的订单数；
- 可选：重复购买客户的商品金额代理指标。

**建议方法：**

1. 订单表连接客户表；
2. 筛选已完成订单；
3. 按 `customer_unique_id` 统计去重订单数；
4. 用 `CASE WHEN` 区分一次购买与多次购买客户；
5. 再汇总客户分层结果。

**你需要能解释：**

- 为什么不能用 `customer_id` 判断复购；
- 为什么这个结果仅表示这份数据时间范围内的重复购买；
- 为什么它不等于严格定义的用户留存率。

#### 3.4 Q4：州级区域差异

**目的：** 用区域维度展示多指标综合分析能力。

**需要计算：**

- 州；
- 已完成订单数；
- 商品金额代理指标；
- 平均订单商品金额；
- 平均配送天数；
- 平均评分或低评分率。

**建议方法：**

- 客户所在州作为订单归属区域；
- 先计算订单级配送指标，再按州汇总；
- 对订单数很少的州，要谨慎解释平均值。

**你需要能解释：**

- 为什么订单少的州平均值可能不稳定；
- 为什么州级差异只能说明样本表现，不一定能说明物流原因；
- 为什么不能直接将客户州和卖家州的差异当成真实运输距离。

#### 3.5 Q5：配送表现与延迟情况

**目的：** 计算数据中最容易理解、也容易展示的服务体验指标。

**需要计算：**

- 已完成订单的配送时长；
- 配送时长的平均值、中位数或分位数（中位数可选）；
- 延迟订单数；
- 延迟率；
- 延迟率较高的州或品类（至少选一个维度）。

**建议方法：**

```text
配送时长 = order_delivered_customer_date - order_purchase_timestamp
延迟订单 = order_delivered_customer_date > order_estimated_delivery_date
```

**你需要能解释：**

- 为什么只用有完整时间字段的已完成订单；
- 为什么平均值可能受到极端慢订单影响；
- 为什么“延迟”是相对预计送达时间，而不是相对下单时间。

#### 3.6 Q6：配送与评分的关系

**目的：** 展示你会把两个业务维度做合理对比，但不做无依据因果判断。

**最低分析设计：**

将订单按是否延迟分为两组，比较：

- 平均评分；
- 低评分率；
- 有效评价数量。

**可选扩展：**

将配送时长划分为 0—3 天、4—7 天、8—14 天、15 天以上，比较每组的低评分率。

**你需要能解释：**

- 这个分析只能说明关联；
- 评分还可能受到商品质量、价格、客服等未记录因素影响；
- 评价缺失订单不能纳入评分分母。

### SQL 技能清单

核心版 SQL 至少覆盖：

| SQL 能力 | 本项目中的使用位置 |
|---|---|
| `SELECT` / `WHERE` | 筛选已完成订单、有效评价 |
| `GROUP BY` / 聚合函数 | 按月、品类、州汇总 |
| `JOIN` | 订单、客户、商品、评价等连接 |
| `COUNT(DISTINCT ...)` | 防止订单数量重复计算 |
| `CASE WHEN` | 客户分层、是否延迟、低评分标记 |
| 日期函数 | 提取月份、计算配送天数 |
| CTE | 将复杂逻辑拆成更容易理解的步骤 |
| 窗口函数 | 可选，不强制；用于排行或更复杂对比 |

### Dashboard 制作步骤

1. 确认每个 KPI 的 SQL/Python 数值；
2. 导出 Tableau 需要的汇总数据；

### “业务建议”应写到什么程度

你不需要写“年度战略”。只需要根据结果写可验证、不过度夸张的建议，例如：

## 11. 项目工作规则

- 解释 SQL/Python 概念；
